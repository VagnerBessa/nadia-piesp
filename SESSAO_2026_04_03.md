## 1. Entendimento da Arquitetura de Busca

### Por que RAG e contexto longo não foram possíveis

A sessão começou com a pergunta: como o sistema faz busca na base de dados, dado que RAG tradicional e leitura de contexto longo foram descartados?

**RAG** falha porque embeddings não preservam valores numéricos com precisão — um modelo de embedding não garante que "R$ 500 milhões em Campinas em 2024" seja recuperado com exatidão em uma busca vetorial.

**Contexto longo** falha por dois motivos documentados na sessão anterior (`README_TEST_CONCLUSION.md`):
1. Limite de payload WebSocket no browser (Chrome recusa frames grandes)
2. "Colapso de atenção" — com 5.000+ linhas de CSV denso, o modelo alucina dados

### A solução: Function Calling + Filtragem determinística em JavaScript

O CSV inteiro é carregado em memória no browser (via `import ... ?raw` do Vite). Quando o modelo precisa de dados, chama uma ferramenta — que é uma função JavaScript pura rodando no browser do usuário:

```
Usuário pergunta
    ↓
Gemini decide qual ferramenta chamar e com quais argumentos
    ↓
JavaScript filtra o CSV linha a linha (O(n), < 100ms)
    ↓
Retorna JSON com top 10 + total
    ↓
Gemini formula a resposta em linguagem natural
```

**Resultado:** precisão 100% determinística, sem alucinação de dados, sem infraestrutura de servidor.

### Regra de ouro da arquitetura (derivada das sessões)

| Tipo de conteúdo | Estratégia | Por quê |
|---|---|---|
| Texto narrativo (metodologia, regras) | Contexto longo (systemInstruction) | LLMs compreendem prosa com excelência |
| Dados tabulares / CSV | Function Calling (Tools) | LLMs falham em filtrar e agregar linhas numéricas densas |
| Dados pequenos (dicionário ~2KB) | Contexto longo | Volume insignificante, sem risco de diluição |

`piespDataService.ts` aceitava apenas dois filtros:

```typescript
interface FiltroPiesp {
  ano?: string;       // match exato
  municipio?: string; // substring case-insensitive
}
```

### Estado atual (após esta sessão)

```typescript
interface FiltroPiesp {
  ano?: string;       // match exato
  municipio?: string; // substring
  regiao?: string;    // substring — ex: "RA Campinas", "RMSP"

  setor?: string;     // substring — ex: "automotivo", "energia"
```

## 3. Busca Semântica via Campo `descricao`

### O problema

A limitação de busca por substring é que conceitos semânticos amplos não têm correspondência direta nos campos estruturados. "Energia limpa" não encontra registros com CNAE de energia solar se o campo `setor_desc` diz "Energia Elétrica" e não "energia limpa".

### A solução em duas partes

**Parte 1 — Filtro `descricao` com lógica OR por vírgula**

O campo `descr_investimento` contém texto livre (ex: "Implantação de usina solar fotovoltaica..."). Adicionado como filtro com suporte a múltiplos termos separados por vírgula — qualquer match retorna a linha:

```typescript
if (filtro.descricao) {
  const termos = filtro.descricao.split(',').map(t => t.trim().toLowerCase()).filter(Boolean);
  if (!termos.some(t => descricaoLinha.includes(t))) match = false;
}
```

Isso mantém **uma única chamada à ferramenta**, sem round-trips extras ao Gemini (o que preserva a latência).

**Parte 2 — Instrução de expansão semântica no system prompt**

O modelo é instruído a traduzir conceitos amplos em termos específicos e passá-los separados por vírgula:

```
"energia limpa" → descricao: "solar,eólic,fotovoltaic,biogás,biomassa,hidrogênio"
"tecnologia"    → descricao: "data center,inteligência artificial,automação,robótica"
```

### Bug encontrado e corrigido via teste

Durante a validação com CSV mockado, identificamos que `"eólica"` não dá match em `"parque eólico"` — a busca é por substring exata e o adjetivo em português varia por gênero (`eólica` vs `eólico`).

**Solução:** usar o radical da palavra, que captura todas as variações:
- `"eólica"` → `"eólic"` (captura "eólico" e "eólica")
- `"fotovoltaica"` → `"fotovoltaic"` (captura ambas as formas)
- `"elétrico"` → `"elétric"` (captura "elétrico", "elétrica", "elétricos")

Os exemplos no system prompt foram atualizados para usar radicais.

**Cobertura de testes:** 16 casos passando — filtros individuais, OR por vírgula, combinações, e ordenação por valor.

## 4. Análise da Arquitetura de Skills

### Como funciona

O `skillDetector.ts` detecta keywords na mensagem do usuário e injeta o conteúdo da skill correspondente no system instruction da chamada. São 9 skills temáticas (~3–4 KB cada):

- Emprego e Empregabilidade
- Qualificação Profissional
- Logística e Infraestrutura
- Inovação e Tecnologia
- Desenvolvimento Regional
- Cadeias Produtivas
- Transição Energética e Sustentabilidade
- Comércio Exterior e Exportações
- Inteligência Empresarial

A skill com maior score de keywords vence. A normalização remove acentos antes da comparação.

### O que está bem

- Detecção por score é robusta para o volume de keywords cadastrado
- Injeção cirúrgica por mensagem — não acumula no histórico
- `inteligencia_empresarial` tem tratamento especial: troca function calling por Google Search e autoriza uso do conhecimento geral do modelo sobre a empresa

### Limitação identificada: skills não chegam à voz

No **chat** (`useChat.ts`), `buildSystemInstructionWithSkill()` é chamada a cada mensagem — a skill é injetada na instrução daquela chamada específica. ✅

Na **voz** (`useLiveConnection.ts`), o `systemInstruction` é definido **uma vez na abertura da conexão WebSocket** e não pode ser alterado durante a sessão. A API Gemini Live não suporta reinjeção de instrução por turno. ❌

**Consequência:** perguntas sobre empregabilidade, logística ou transição energética na interface de voz não recebem a lente analítica especializada.

**Opções avaliadas:**

| Opção | Trade-off |
|---|---|
| Incluir todas as skills na instrução inicial da voz | ~30KB a mais no contexto, instruções genéricas sem foco no tema da conversa |
| Reconectar sessão ao detectar keyword | Latência perceptível, quebra de experiência |

Nenhuma opção é ideal. A limitação é arquitetural do Gemini Live, não da implementação local. Registrado para decisão futura.
